ความเร็วของหน้าเว็บส่งผลโดยตรงต่อการจัดอันดับ SEO และประสบการณ์ผู้ใช้ เรียนรู้วิธีเปลี่ยนเว็บหน่วงให้ลื่นไหลทะลุนรกด้วย Core Web Vitals

First Contentful Paint (FCP)

≤ 1.8s ดี 1.8s - 3s ควรปรับปรุง > 3s แย่

มันคืออะไร?

เวลาที่เบราว์เซอร์ใช้ในการแสดงผลเนื้อหาชิ้นแรกสุดจาก DOM เช่น ข้อความ รูปภาพ หรือปุ่มตัวแรกสุดหลังจากกดเข้าเว็บ

❌ ตัวอย่างที่แย่ (ฝันร้ายของผู้ใช้)

หน้าเว็บผูกมัดไฟล์ CSS หรือ JavaScript ขนาดใหญ่หนักอึ้งไว้ที่ Header เบราว์เซอร์ต้องรอโหลดจนเสร็จ ทำให้ผู้ใช้จ้องหน้าจอขาวว่างเปล่า (Blank Screen) นานเกิน 3 วินาที

🛡️ วิธีแก้ไขและแนวทางปฏิบัติ

  • ทำ Critical CSS แยกโค้ดสไตล์ส่วนบนสุดมาโหลดก่อน
  • ใช้คำสั่ง Defer หรือ Async ปล่อยให้ JavaScript โหลดทีหลังไม่ให้บล็อกการวาดหน้าจอ

Largest Contentful Paint (LCP) Core Web Vitals

≤ 2.5s ดี 2.5s - 4s ควรปรับปรุง > 4s แย่

มันคืออะไร?

เวลาที่ใช้ในการแสดงผลองค์ประกอบหลักที่มีขนาด "ใหญ่ที่สุด" ในส่วนที่ผู้ใช้มองเห็นครั้งแรก (เช่น รูปภาพแบนเนอร์ Hero Image หรือกล่องข้อความพาดหัวหลัก)

❌ ตัวอย่างที่แย่

ใช้ภาพแบนเนอร์ความละเอียดสูงสัดส่วนไม่ได้บีบอัดไฟล์ล่อไป 5MB แถมโหลดผ่านคำสั่ง JavaScript ตัวท้ายๆ ทำให้อ่านตัวหนังสือจบแล้วรูปใหญ่ตรงกลางก็ยังไม่ขึ้น

🛡️ วิธีแก้ไขและแนวทางปฏิบัติ

  • แปลงรูปภาพเป็นฟอร์แมตยุคใหม่ เช่น WebP / AVIF และบีบอัดให้เหมาะสม
  • ใส่แท็ก <link rel="preload"> ที่รูปภาพหลักเพื่อสั่งเบราว์เซอร์ให้ดึงภาพทันทีตั้งแต่เริ่ม

Interaction to Next Paint (INP) Core Web Vitals

≤ 200ms ดี 200ms - 500ms ปรับปรุง > 500ms แย่

มันคืออะไร?

ตัวชี้วัดความลื่นไหลในการตอบสนองของหน้าเว็บ โดยวัดเวลาตั้งแต่ผู้ใช้คลิกปุ่ม แตะลิงก์ หรือพิมพ์งาน ไปจนถึงขั้นตอนที่หน้าจอกลับมาอัปเดตตอบสนองภาพถัดไป

❌ ตัวอย่างที่แย่

กดปุ่ม 'หยิบใส่ตะกร้า' แล้วระบบค้างนิ่งสนิทไป 1 วินาที เพราะสคริปต์เบื้องหลังกำลังทำงานประมวลผลหนักหน่วงขวางทาง Main Thread ทำให้ผู้ใช้หงุดหงิดกดซ้ำรัวๆ

🛡️ วิธีแก้ไขและแนวทางปฏิบัติ

  • ใช้โครงสร้าง API ใหม่ scheduler.yield() เพื่อแบ่งย่อยงานสคริปต์ก้อนใหญ่ให้เบราว์เซอร์หายใจคล่องขึ้น
  • หากงานหนักจริง ให้เขียนผลลัพธ์ด่วนด้วยการแสดง Loading Spinner ทันทีที่คลิก

Cumulative Layout Shift (CLS) Core Web Vitals

≤ 0.1 ดี 0.1 - 0.25 ปรับปรุง > 0.25 แย่

มันคืออะไร?

การวัดความเสถียรของหน้าเว็บ ตรวจสอบว่าองค์ประกอบต่างๆ มีการขยับ เลื่อน หรือกระโดดไปมาอย่างกะทันหันระหว่างที่โหลดหน้าเว็บหรือไม่

❌ ตัวอย่างที่แย่

ผู้ใช้กำลังจะเอานิ้วกดปุ่ม "ยกเลิก" แต่อยู่ดีๆ แบนเนอร์โฆษณาด้านบนเพิ่งโหลดเสร็จ ดันเนื้อหาลงมาข้างล่างฉับพลัน ทำให้กดผิดไปโดนปุ่ม "ยืนยันการจ่ายเงิน" แทน

🛡️ วิธีแก้ไขและแนวทางปฏิบัติ

  • ระบุขนาดกว้างและสูงที่แน่นอนให้กับรูปภาพและกรอบโฆษณาเสมอ เช่น width="" height="" ใน HTML
  • จองพื้นที่ (Aspect Ratio Boxes) รอไว้ เพื่อไม่ให้เนื้อหากระเด้งเมื่อโหลดเสร็จ

Total Blocking Time (TBT)

≤ 200ms ดี 200ms - 600ms ปรับปรุง > 600ms แย่

มันคืออะไร?

เวลารวมที่เบราว์เซอร์โดน "บล็อก" โดยสคริปต์งานเดี่ยวที่ประมวลผลยาวนานเกิน 50ms (Long Tasks) ในระหว่างหน้าเว็บกำลังโหลด ส่งผลกระทบโดยตรงต่อความหน่วง

❌ ตัวอย่างที่แย่

หน้าเว็บยัดไฟล์ปลั๊กอิน สคริปต์สถิติแทร็กพฤติกรรมจากภายนอก (Third-party) เยอะจนล้น หน้าเว็บขึ้นโครงปุ่มครบหมดแล้ว แต่ผู้ใช้นิ้วเลื่อนหน้าจอไม่ได้ เคาะเมนูเท่าไหร่ก็แข็งทื่อ

🛡️ วิธีแก้ไขและแนวทางปฏิบัติ

  • ตัดสคริปต์ขยะที่ไม่ได้ใช้งานทิ้ง ลด ละ การพึ่งพา Third-party scripts
  • ใช้เทคโนโลยี Web Workers แยกงานประมวลผลหนักๆ ไปรันฉากหลัง ไม่ขัดขวาง Main Thread

Speed Index (SI)

≤ 3.4s ดี 3.4s - 5.8s ปรับปรุง > 5.8s แย่

มันคืออะไร?

ดัชนีภาพรวมความเร็วความพร้อมที่วัดว่าเนื้อหาทั้งหมดในหน้าจอแรก (Above the fold) ค่อยๆ โผล่แสดงผลสมบูรณ์เต็มตาได้ลื่นไหลเรียบร้อยเร็วเพียงใด

❌ ตัวอย่างที่แย่

ส่วนประกอบหน้าต่างกระจัดกระจาย โหลดกล่องสีเหลี่ยมขึ้นมาก่อน ทิ้งช่วงนิ่ง แล้วปุ่มค่อยกระตุกตามมาทีหลัง โหลดติดขัดสะดุดสายตา

🛡️ วิธีแก้ไขและแนวทางปฏิบัติ

  • จัดลำดับความสำคัญของโครงสร้าง จัดส่งเฉพาะข้อมูลที่จะฉายในจอแรกให้ไวที่สุด
  • เปิดใช้งานโครงสร้าง Lazy Loading กับรูปภาพหรือวิดเจ็ตที่อยู่ครึ่งล่างของเว็บลงไปเพื่อไม่แย่งแบนด์วิดท์ชิ้นแรก
